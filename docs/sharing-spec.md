# Monostone · 分享功能设计规范

长录音卡片（以及未来其他类型）的"分享"功能如何工作。

**版本**：v0.1 · 2026-04-09

---

## 1. 为什么要做分享 · 产品决策

用户在 Monostone 里沉淀的内容（会议总结、研究报告、灵感笔记）有两种去向：

1. **留在 Monostone 里** — 作为自己的 context 继续复利
2. **分享出去** — 给团队、给投资人、给自己的其他工具（Notion / Obsidian）

分享不是"导出逃离"，而是**协作的延伸**。所以分享出去的内容必须：

- 足够结构化，收到的人一眼能看懂
- 足够 portable，换任何工具都能打开
- 带上必要的 metadata（时间、参与人、项目），让对方知道这是什么
- **不泄露上下文** — 只分享这一张卡片的内容，不夹带 Memory 或其他卡片的私人信息

---

## 2. 哪些卡片可以分享

| 类型 | 是否支持 | 为什么 |
|---|---|---|
| **长录音** | ✅ **v0.1 先做这个** | 会议总结是最强的分享需求 |
| **指令** | v0.2 再加 | 邮件草稿的"分享"其实就是"发送"，不需要另一层分享 |
| **灵感** | v0.2 再加 | 需求弱，灵感大多是私人思考 |
| **待办/日程** | ❌ 不做 | 这些已经在外部日历/Linear 里了 |

---

## 3. 分享的内容是什么

### 3.1 UI 分成两层

**第一层：结构化摘要**（卡片详情页直接看的）
- 标题 + meta
- 参与人
- 4 条核心 bullet
- Action Items
- 决策
- 记忆归因
- 完整转录（折叠）

**第二层：完整总结**（点"查看完整总结"弹出的 modal）
- 和第一层不同 —— 这是一个 **narrative 叙述版本**，由 LLM 从转录生成
- 包含：核心议题段 / 关键讨论段 / 决策列表 / Action Items 列表 / 后续计划段
- 适合直接复制到邮件或 Notion

### 3.2 分享导出的内容 = 完整总结

点"分享"按钮 → 分享 sheet 打开 → 预览和导出的都是**第二层的完整总结**。

原因：
- 第一层太简短（只有 bullet），分享出去信息量不够
- 第一层包含"本次会议学到了什么"这种私人 memory 归因，**不应分享给外人**
- 第二层是专门为分享打磨的 narrative，对外友好

**⚠️ 关键决策**：分享出去的内容永远 **不包括 memory 归因 section**，也 **不包括完整原始转录**。只包括用户主动想分享的部分。

## 4. 生成完整总结的 LLM Prompt

### Prompt 模板

```
You are generating a shareable meeting summary from a raw transcript.
The summary will be exported to Markdown and sent to external collaborators.

## Input
- Meeting title: {title}
- Participants: {participants}
- Duration: {duration}
- Date: {date}
- Project: {project}
- Raw transcript: {full_transcript}

## Output requirements

Write in the same language as the transcript (Chinese or English).
Structure the output as the following sections in this exact order:

1. **核心议题 / Core Topic** (1 paragraph, 60-120 words)
   - What was this meeting about at the highest level?
   - Who drove the discussion?

2. **关键讨论 / Key Discussion** (2-4 paragraphs)
   - Break down by topic, not by time
   - Use <b>...</b> to bold key terms
   - Each paragraph covers one distinct thread
   - Include specific numbers, names, references when mentioned

3. **决策 / Decisions** (bullet list)
   - Format: "[decision] ([who made it])"
   - Only include decisions that were explicitly made, not suggested

4. **Action Items** (bullet list)
   - Format: "<b>[owner]</b> · [task] · [deadline]"
   - Only include items with clear owner AND deadline

5. **后续 / Next Steps** (1 paragraph, optional)
   - What's the next touchpoint?
   - What needs to happen before then?

## Constraints

- DO NOT include personal memory inferences ("Monostone learned that X likes Y")
- DO NOT include the raw transcript
- DO NOT invent details not in the transcript
- DO NOT add a "generated by" footer (this is added by the export layer)
- Keep it professional but not stiff
- Use the user's natural writing voice (learned from their email style)

Output as JSON:
{
  "title": "...",
  "meta": "...",
  "sections": [
    { "h": "核心议题", "p": "..." },
    { "p": "..." },  // additional paragraphs under prev heading
    { "h": "决策", "ul": ["...", "..."] },
    ...
  ]
}
```

### 为什么用 JSON 而不是直接 Markdown

1. 前端需要灵活的渲染（段落 vs 列表 vs 标题）
2. 后端可以做结构化校验（必须有哪些 section）
3. 导出时转成不同格式（md / pdf / txt）统一从 JSON 出发
4. 未来可以做"按 section 筛选分享"（用户可能只想分享 Action Items）

## 5. 导出格式

用户在分享 sheet 里可以选 3 种格式：

### 5.1 Markdown（默认、推荐）

**场景**：发给团队、Notion / Obsidian 用户、技术向收件人

**输出示例**：
```markdown
# 和蔡哥的 Series A 跟进会

> 2026-04-09 10:30 · 42:18 · 蔡哥 / 明明 / 郑灿 / 马俊

## 核心议题

本次会议是 Series A D-Day 的深度对接。蔡哥带领 Linear Capital 团队就 Monostone 的产品定位、竞争壁垒、GTM 节奏、硬件执行风险四个维度做了 42 分钟的系统性质询……

## 关键讨论

**关于定位**：Linear 明确反对把 Monostone 定位成 "AI infra"……

**关于竞争**：Linear 把 OpenAI 排在威胁榜首……

## 决策

- 不做 Kickstarter，坚持独立站 DTC 首发（明明主张，Linear 支持）
- 企业场景优先级高于家庭场景，作为 B2B 扩展路径（Linear 建议，待评估）

## Action Items

- **Marshall** · 锁定 ODM 合作方 · 4 月底
- **石慧** · 把双麦 SNR 数据整理成 marketing 故事 · 本周
- **明明** · 研究 EO 14117 美国数据法规 · 下周前

---
*由 Monostone 生成 · 2026-04-09*
```

**技术实现**：
- 前端直接用 `buildShareMarkdown(cardId)` JS 函数从 JSON 转换
- 函数已在原型里实现，见 index.html

### 5.2 PDF

**场景**：发给不用 Markdown 的人、需要存档、打印

**实现策略**：
- **方案 A**（推荐）：前端把 Markdown 渲染成 HTML，然后用浏览器的 `window.print()` + CSS 打印样式 → 存为 PDF。零后端依赖。
- **方案 B**：后端用 puppeteer / playwright 把 HTML 渲染成 PDF。更可控但需要服务。

对 iOS 原生 App：用系统 `UIPrintInteractionController` 或 `PDFKit` 把 HTML 转 PDF 存到 Files。

**PDF 样式**（和 Web 预览一致但加了 print 友好的 CSS）：
- A4 页面
- 14pt 正文字号
- 黑底白字改成白底黑字（打印友好）
- 链接保留但以灰色显示（省墨）
- 页脚加 URL 和页码

### 5.3 纯文本（.txt）

**场景**：老式邮件、粘贴到 Slack/微信等不支持 Markdown 的地方

**转换规则**：
- `# H1` → 全大写标题 + 下划线 `===`
- `## H2` → 大写 + `---`
- `**bold**` → 去掉（不支持）
- `[link](url)` → `link (url)`
- `- item` → `• item`
- 每段之间空行

## 6. 分享的"目的地"

分享 sheet 底部的 8 个 destination（原型里已实现）：

| 图标 | 名称 | 行为 | 实际实现 |
|---|---|---|---|
| ⎘ | 复制 | 复制到剪贴板 | `navigator.clipboard.writeText(content)` |
| M | 邮件 | 打开系统邮件 + 预填 | `mailto:?subject=...&body=...` |
| i | 信息 | iMessage / SMS | iOS: `sms:&body=...` |
| N | Notion | 通过已连接的 Notion integration 写入指定 page | 调用 Notion API |
| S | Slack | 发送到已连接的 Slack channel | 调用 Slack API |
| W | 微信 | 调用微信 SDK 分享 | iOS 专用，用 WeChat Share SDK |
| ↓ | AirDrop | 系统 share sheet | iOS `UIActivityViewController` |
| □ | 文件 | 存到 Files App | iOS `UIDocumentPickerViewController` |

**iOS 端的实际实现**：直接用系统 `UIActivityViewController`，它会自动列出所有支持的目的地（用户装了什么 app 就显示什么）。上面的 8 个图标只是 web demo 的视觉模拟。

**web demo 的实现**：
- 复制 → `navigator.clipboard.writeText()`
- 邮件 → `window.open('mailto:...')`
- 其他 → 目前是 toast 提示 "已打开 XX"，实际对接需要各自的 integration

## 7. 后端 API

### 7.1 获取完整总结

```
GET /cards/{id}/full-summary
Response: {
  title, meta, sections: [ {h?, p?, ul?}, ... ]
}
```

后端流程：
1. 检查 `card.full_summary_cached` 是否存在
2. 存在 → 直接返回
3. 不存在 → 调 LLM 生成 → 存 `card.full_summary_cached` → 返回

**缓存策略**：完整总结一旦生成就缓存，除非用户主动让 Agent 重写（通过 chat：`"把这个会议总结重新写一下"`）。

### 7.2 导出

```
GET /cards/{id}/export?format=md|pdf|txt
```

后端流程：
1. 拉取 full_summary（走上面的 cache）
2. 根据 format 转换
3. Markdown / TXT：直接返回 text/markdown 或 text/plain
4. PDF：返回 `application/pdf` binary

**限制**：
- PDF 生成有 rate limit（10/min/user）
- 导出有 audit log（什么时候导了哪张卡到什么格式）

### 7.3 分享到第三方

```
POST /cards/{id}/share
body: {
  destination: 'notion' | 'slack' | 'email' | ...,
  params: { ... }  // destination-specific
}
```

这个 API 会：
1. 验证用户有这个 destination 的 integration
2. 构造 payload（格式和目标平台匹配）
3. 调外部 API 推送
4. 返回成功/失败 + external_url（如果有）

**⚠️ 重要**：分享到 Notion / Slack 等 destination 时，**不是在后端生成 markdown 再 POST 给对方**，而是按对方的原生格式构造（Notion blocks / Slack blocks）。这样效果最好。

## 8. 隐私与控制

### 8.1 默认不包含

分享内容 **默认不包含**：
- 原始音频 URL
- 完整转录文本
- Memory 归因（"本次会议学到了什么"）
- 相关卡片/项目的 cross-link
- 用户的 chat with Agent 历史

### 8.2 用户可选添加

在分享 sheet 里可以加一个 toggle（未来版本）："附加完整转录"。默认 off。

### 8.3 审计

每次分享记录：
```sql
INSERT INTO share_log (
  card_id, user_id, destination, format, recipient, shared_at
)
```

用户在"隐私与数据"里可以看到自己的所有分享历史。
